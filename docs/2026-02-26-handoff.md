## Context Delivery Fix (completed this session)

### Root Cause

Skills ARE the intentional routing mechanism ‚Äî not a bug. The assembler matches skills by entity name pattern and loads their linked context docs (primary/reference/supplementary). The issue was simpler: `generate-mapping-skills.ts` had a **stale hardcoded workspace ID** (`fbc37e23...`) that didn't match the actual DB workspace (`2ac4e497...`), so the script had never successfully run.

### What Was Fixed

**1. `scripts/generate-mapping-skills.ts`** ‚Äî workspace ID now reads dynamically from DB (`SELECT id FROM workspace LIMIT 1`). Script was run ‚Üí **54 skills created, 262 context links**.

Each skill bundles for one entity:
- PRIMARY: VDS entity skill doc + relevant SM source tables
- REFERENCE: SM domain context + VDS category overview + enum maps
- SUPPLEMENTARY: Relevant regulatory mortgage docs

**2. `src/lib/generation/context-assembler.ts`** ‚Äî added two direct inclusion paths that bypass skill routing, since these don't need a skill to gatekeep them:
- **Entity Knowledge** (`subcategory='entity_knowledge'`, by `entityId`) ‚Üí reference priority. This is the fast-loop feedback path ‚Äî corrections and resolved questions flow here regardless of skill state.
- **Foundational docs** (`category='foundational'`) ‚Üí supplementary priority, budget-capped. Distilled learnings + 82 mortgage servicing docs always flow to every generation.

`assembleContext()` signature now accepts optional `entityId` param.

**3. `src/lib/generation/runner.ts`** ‚Äî passes `targetEntity.id` to `assembleContext`. Fixed misleading comment that referenced "learnings query" ‚Äî now correctly notes learnings are surfaced via Entity Knowledge context docs included directly by entityId.

**4. `src/lib/generation/context-cache.ts`** ‚Äî `entityId` added to the cache key (`makeKey`) and threaded through `getCachedContext`/`setCachedContext` signatures. Without this, different entities with the same name pattern could return stale cached context missing Entity Knowledge docs.

### Verified State

For `loss_mitigation_loan_modification`, a generation now receives:
- **18 skill-linked contexts** ‚Äî VDS entity doc, SM Step + LoanInfo tables, SM Loss Mitigation domain, CFPB regulatory docs (the `Mapping: Loan` skill also matches via substring ‚Äî harmless, adds useful cross-entity context)

Previously: zero context sections in the prompt.

### Existing Entity Knowledge Infrastructure (works, now connected)

**`src/lib/generation/mapping-learning.ts`** ‚Äî When a reviewer edits a mapping (PATCH), `extractMappingLearning()` detects significant changes (source entity/field/transform changed), creates a `learning` record, then calls `rebuildEntityKnowledge()`.

**`src/lib/generation/entity-knowledge.ts`** ‚Äî `rebuildEntityKnowledge()` aggregates all learnings + resolved questions + resolved comment threads for an entity into a single markdown context doc. Upserts into `context` table (category="adhoc", subcategory="entity_knowledge", entityId set). Now directly fetched by the assembler via entityId ‚Äî no skill dependency.

**Question resolution** (`src/app/api/.../questions/[id]/resolve/route.ts`) ‚Äî When a question is resolved, it creates a learning record AND calls `rebuildEntityKnowledge()`, so Q&A flows into Entity Knowledge immediately.

## Key Architecture Notes

### Generation Flow
```
UI: Start Batch Run ‚Üí POST /batch-runs
  ‚Üí batch-runner.ts: per entity:
    ‚Üí runner.ts: startGeneration() [sync: load fields, assemble context, build prompt]
    ‚Üí runner.ts: executeGeneration() [async: call Claude, parse YAML, validate, persist]
    ‚Üí batch-runner.ts: persist fieldMapping rows + create questions
```

### Context Assembly Flow (fixed)
```
assembleContext(workspaceId, entityName, tokenBudget, ..., entityId)
  ‚Üí matchSkills(workspaceId, entityName)  // matches entity-specific + broad skills
  ‚Üí load skill-linked contexts            // VDS doc, SM tables, domain, regulatory
  ‚Üí [NEW] Entity Knowledge by entityId   // direct lookup, no skill needed (reference)
  ‚Üí [NEW] Foundational docs              // always included (supplementary, budget-capped)
  ‚Üí auto-include enum contexts           // works if source tables in prompt
  ‚Üí FTS5 query-aware retrieval           // only if query param provided
  ‚Üí trim to token budget
```

### Feedback Flow (fully connected)
```
Reviewer edits mapping ‚Üí PATCH /mappings/[id]
  ‚Üí extractMappingLearning(old, new)
    ‚Üí insert learning (scope="field", source="review")
    ‚Üí rebuildEntityKnowledge()
      ‚Üí gather learnings + questions + threads
      ‚Üí render markdown
      ‚Üí upsert context doc (subcategory="entity_knowledge", entityId set)
      ‚Üí linkToMatchingSkills()  // links to entity skill (bonus, not required)
      ‚Üì
  Next generation ‚Üí assembleContext() ‚Üí direct entityId lookup ‚Üí correction in prompt
```

### Important Discovery: executeGeneration vs batch-runner

`executeGeneration()` (in `runner.ts`) only:
- Calls the LLM
- Parses the YAML output
- Updates the `generation` record (status, outputParsed, tokens)

It does NOT persist `fieldMapping` rows. That's handled by `batch-runner.ts:saveMappingsAndQuestions()`, which iterates `outputParsed.fieldMappings` and inserts rows. CLI scripts that bypass batch-runner need to persist mappings themselves (as `persist-and-eval.ts` does).
