## Context Delivery Fix (completed this session)

### Root Cause

Skills ARE the intentional routing mechanism — not a bug. The assembler matches skills by entity name pattern and loads their linked context docs (primary/reference/supplementary). The issue was simpler: `generate-mapping-skills.ts` had a **stale hardcoded workspace ID** (`fbc37e23...`) that didn't match the actual DB workspace (`2ac4e497...`), so the script had never successfully run.

### What Was Fixed

**1. `scripts/generate-mapping-skills.ts`** — workspace ID now reads dynamically from DB (`SELECT id FROM workspace LIMIT 1`). Script was run → **54 skills created, 262 context links**.

Each skill bundles for one entity:
- PRIMARY: VDS entity skill doc + relevant SM source tables
- REFERENCE: SM domain context + VDS category overview + enum maps
- SUPPLEMENTARY: Relevant regulatory mortgage docs

**2. `src/lib/generation/context-assembler.ts`** — added two direct inclusion paths that bypass skill routing, since these don't need a skill to gatekeep them:
- **Entity Knowledge** (`subcategory='entity_knowledge'`, by `entityId`) → reference priority. This is the fast-loop feedback path — corrections and resolved questions flow here regardless of skill state.
- **Foundational docs** (`category='foundational'`) → supplementary priority, budget-capped. Distilled learnings + 82 mortgage servicing docs always flow to every generation.

`assembleContext()` signature now accepts optional `entityId` param.

**3. `src/lib/generation/runner.ts`** — passes `targetEntity.id` to `assembleContext`. Fixed misleading comment that referenced "learnings query" — now correctly notes learnings are surfaced via Entity Knowledge context docs included directly by entityId.

**4. `src/lib/generation/context-cache.ts`** — `entityId` added to the cache key (`makeKey`) and threaded through `getCachedContext`/`setCachedContext` signatures. Without this, different entities with the same name pattern could return stale cached context missing Entity Knowledge docs.

### Verified State

For `loss_mitigation_loan_modification`, a generation now receives:
- **18 skill-linked contexts** — VDS entity doc, SM Step + LoanInfo tables, SM Loss Mitigation domain, CFPB regulatory docs (the `Mapping: Loan` skill also matches via substring — harmless, adds useful cross-entity context)

### Existing Entity Knowledge Infrastructure (works, now connected)

**`src/lib/generation/mapping-learning.ts`** — When a reviewer edits a mapping (PATCH), `extractMappingLearning()` detects significant changes (source entity/field/transform changed), creates a `learning` record, then calls `rebuildEntityKnowledge()`.

**`src/lib/generation/entity-knowledge.ts`** — `rebuildEntityKnowledge()` aggregates all learnings + resolved questions + resolved comment threads for an entity into a single markdown context doc. Upserts into `context` table (category="adhoc", subcategory="entity_knowledge", entityId set). Now directly fetched by the assembler via entityId — no skill dependency.

**Question resolution** (`src/app/api/.../questions/[id]/resolve/route.ts`) — When a question is resolved, it creates a learning record AND calls `rebuildEntityKnowledge()`, so Q&A flows into Entity Knowledge immediately.

## Key Architecture Notes

### Generation Flow
```
UI: Start Batch Run → POST /batch-runs
  → batch-runner.ts: per entity:
    → runner.ts: startGeneration() [sync: load fields, assemble context, build prompt]
    → runner.ts: executeGeneration() [async: call Claude, parse YAML, validate, persist]
    → batch-runner.ts: persist fieldMapping rows + create questions
```

### Context Assembly Flow (fixed)
```
assembleContext(workspaceId, entityName, tokenBudget, ..., entityId)
  → matchSkills(workspaceId, entityName)  // matches entity-specific + broad skills
  → load skill-linked contexts            // VDS doc, SM tables, domain, regulatory
  → [NEW] Entity Knowledge by entityId   // direct lookup, no skill needed (reference)
  → [NEW] Foundational docs              // always included (supplementary, budget-capped)
  → auto-include enum contexts           // works if source tables in prompt
  → FTS5 query-aware retrieval           // only if query param provided
  → trim to token budget
```

### Feedback Flow (fully connected)
```
Reviewer edits mapping → PATCH /mappings/[id]
  → extractMappingLearning(old, new)
    → insert learning (scope="field", source="review")
    → rebuildEntityKnowledge()
      → gather learnings + questions + threads
      → render markdown
      → upsert context doc (subcategory="entity_knowledge", entityId set)
      → linkToMatchingSkills()  // links to entity skill (bonus, not required)
      ↓
  Next generation → assembleContext() → direct entityId lookup → correction in prompt
```

### DB Schema Highlights
- `entity` — side="target"|"source", name, workspaceId
- `field` — entityId, name, dataType, milestone
- `fieldMapping` — targetFieldId, sourceEntityId, sourceFieldId, transform, status, confidence, isLatest (copy-on-write)
- `context` — name, category, subcategory, entityId, content, isActive, tags (JSON)
- `skill` — name, applicability (entityPatterns/fieldPatterns), isActive
- `skillContext` — skillId→contextId with role (primary/reference/supplementary)
- `learning` — scope (field/entity/workspace), content, source (review/training/manual)
- `sotEvaluation` — entityId, sourceExactPct, sourceLenientPct, fieldResults (JSON)
- `generation` — entityId, status, promptSnapshot, outputParsed, provider, model
